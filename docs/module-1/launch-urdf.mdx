## Short Overview

This chapter delves into the Unified Robot Description Format (URDF), an XML-based format crucial for defining robot structures for use in both ROS 2 and Gazebo. You will learn the fundamental components of URDF, including links and joints, and understand how to create a minimal robot model. The chapter also explains the integration of URDF files within the ROS launch system, covering launch files and parameter management.

## Learning Objectives

By the end of this chapter, you will be able to:

*   Explain the difference between a **link** (body part) and a **joint** (connection) within URDF.
*   Provide a minimal, complete **URDF XML example** for a two-link arm, illustrating fixed and continuous joints.
*   Explain how the URDF file is integrated via the ROS launch system, including Launch Files and Parameter Management.

## Main Sections

### 1. Introduction to URDF: Describing Your Robot

The Unified Robot Description Format (URDF) is an XML format for describing all aspects of a robot. It's used to represent the robot's kinematic and dynamic properties, visual appearance, and collision geometry. URDF files are essential for visualizing your robot in tools like RViz, simulating it in Gazebo, and enabling various ROS 2 packages to interact with its structure.

### 2. Links and Joints: The Building Blocks of a Robot

At its core, a URDF file defines a robot as a collection of **links** connected by **joints**.

#### Links: The Robot's Body Parts

A **link** represents a rigid body part of the robot. Think of it as a segment of the robot's bodyâ€”an arm segment, a wheel, a torso, or a camera housing. Each link has properties such as its name, visual appearance (color, mesh), inertial properties (mass, inertia), and collision geometry (for physics simulation).

#### Joints: Connecting the Parts

A **joint** defines the connection between two links, specifically a *parent* link and a *child* link. Joints specify the type of motion allowed between these links (e.g., revolute, prismatic, fixed) and their kinematic properties (axis of rotation, limits of motion).

*   **Fixed Joint**: No motion allowed between the parent and child link. They move as one rigid body. Useful for connecting sensors or static parts.
*   **Revolute (Continuous) Joint**: Allows rotation around a single axis. A `continuous` joint is a type of revolute joint that has no upper or lower limits (e.g., a wheel that can spin infinitely).

### 3. A Minimal URDF Example: Two-Link Arm

Let's construct a simple URDF for a two-link arm. This example illustrates how links and joints are defined and connected.

```xml
<?xml version="1.0"?>
<robot name="two_link_arm">

  <!-- Base Link -->
  <link name="base_link">
    <visual>
      <geometry>
        <cylinder length="0.1" radius="0.05"/>
      </geometry>
      <material name="blue">
        <color rgba="0 0 0.8 1"/>
      </material>
    </visual>
  </link>

  <!-- Joint 1: Connects base_link to link1 (Revolute/Continuous) -->
  <joint name="joint1" type="continuous">
    <parent link="base_link"/>
    <child link="link1"/>
    <origin xyz="0 0 0.1" rpy="0 0 0"/>
    <axis xyz="0 0 1"/>
  </joint>

  <!-- Link 1 -->
  <link name="link1">
    <visual>
      <geometry>
        <cylinder length="0.2" radius="0.03"/>
      </geometry>
      <material name="green">
        <color rgba="0 0.8 0 1"/>
      </material>
    </visual>
  </link>

  <!-- Joint 2: Connects link1 to link2 (Fixed) -->
  <joint name="joint2" type="fixed">
    <parent link="link1"/>
    <child link="link2"/>
    <origin xyz="0 0 0.2" rpy="0 0 0"/>
  </joint>

  <!-- Link 2 (End Effector) -->
  <link name="link2">
    <visual>
      <geometry>
        <box size="0.05 0.05 0.05"/>
      </geometry>
      <material name="red">
        <color rgba="0.8 0 0 1"/>
      </material>
    </visual>
  </link>

  <!-- Define materials -->
  <material name="blue">
    <color rgba="0 0 0.8 1"/>
  </material>
  <material name="green">
    <color rgba="0 0.8 0 1"/>
  </material>
  <material name="red">
    <color rgba="0.8 0 0 1"/>
  </material>

</robot>
```
**Explanation:**

*   **`<robot name="two_link_arm">`**: The root element, giving the robot a name.
*   **`<link name="base_link">`**: Defines the stationary base of our arm. It has a visual representation (a blue cylinder).
*   **`<joint name="joint1" type="continuous">`**: Connects `base_link` (parent) to `link1` (child). It's a `continuous` joint, allowing full rotation around the Z-axis (`<axis xyz="0 0 1"/>`).
*   **`<link name="link1">`**: The first moving segment, a green cylinder. Its origin `xyz="0 0 0.1"` places it on top of `base_link` visually.
*   **`<joint name="joint2" type="fixed">`**: Connects `link1` to `link2`. It's a `fixed` joint, meaning `link2` is rigidly attached to `link1`.
*   **`<link name="link2">`**: The end-effector, a red box.
*   **`<material>`**: Defines colors for better visualization.

### 4. Integrating URDF with the ROS Launch System

URDF files are typically loaded into ROS 2 via the **ROS launch system**. This system uses **Launch Files** to define how multiple nodes and robot configurations are started and managed.

#### Launch Files: Orchestrating Your Robot

A launch file is an XML (or Python in ROS 2) file that allows you to start and configure multiple ROS 2 nodes and parameters with a single command. To load a URDF, you commonly use the `robot_state_publisher` node, which reads the URDF and publishes the robot's joint states as transformations on the `/tf` topic, allowing visualization in tools like RViz.

**Example Launch File (Python in ROS 2):**

```python
import os
from ament_index_python.packages import get_package_share_directory
from launch import LaunchDescription
from launch_ros.actions import Node

def generate_launch_description():
    # Get URDF file path
    urdf_file_name = 'two_link_arm.urdf'
    urdf_path = os.path.join(
        get_package_share_directory('my_robot_description'), # Replace with your package name
        'urdf',
        urdf_file_name)

    # Robot State Publisher Node
    robot_state_publisher_node = Node(
        package='robot_state_publisher',
        executable='robot_state_publisher',
        name='robot_state_publisher',
        output='screen',
        parameters=[{'robot_description': urdf_path}] # Pass URDF path as a parameter
    )

    # Joint State Publisher Node (for manual joint control/testing)
    joint_state_publisher_node = Node(
        package='joint_state_publisher_gui', # Or 'joint_state_publisher'
        executable='joint_state_publisher_gui',
        name='joint_state_publisher_gui',
        output='screen',
    )

    return LaunchDescription([
        robot_state_publisher_node,
        joint_state_publisher_node
    ])
```

#### Parameter Management with URDF

In the launch file example above, `parameters=[{'robot_description': urdf_path}]` demonstrates **Parameter Management**. The `robot_state_publisher` node is configured with a parameter named `robot_description`, whose value is the path to your URDF file. This is a common way to pass configuration data to ROS 2 nodes, allowing the same node executable to behave differently based on the parameters it receives.

### 5. URDF and Physics Simulation (Gazebo/SDF)

It is important to understand that **URDF itself does not contain physics information**. While URDF defines the robot's geometry, mass, and inertia, it does not specify how these properties interact in a dynamic environment (like friction, restitution, or sensor properties for simulation). For advanced physics simulation in Gazebo, the URDF is often converted or extended into an **SDF (Simulation Description Format)** file, which includes additional tags specific to simulation physics and sensor models. We will explore SDF in Module 2.

## Summary Key Points

*   URDF is an XML format for describing a robot's kinematic, dynamic, and visual properties.
*   Robots are defined by **links** (rigid body parts) connected by **joints** (allowing specified motion).
*   **Fixed** joints create rigid connections, while **continuous** joints allow unlimited rotation.
*   ROS Launch Files are used to load URDF files via nodes like `robot_state_publisher`.
*   **Parameter Management** in launch files allows configuration of nodes, such as passing the URDF path.
*   URDF describes the robot's structure, but **does not specify physics** for simulation (SDF does this).
